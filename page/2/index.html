promis学习

1.promise是什么

MDN给出的定义：Promise对象用于异步计算，一个Promise表示一个现在、将来或永不可能可用的值。
按用途来解释：

  1. 用于异步计算
  2. 可以将异步操作队列化，按照期望的顺序执行，返回符合预期的结果。
  3. 可以在对象之间传递和操作Promise，帮助我们处理队列。

2.为什么会有promise

  1. JavaScript为检查表单而生
  2. 创造它的首要目标是操作DOM
  3. 所以，JavaScript很多操作是异步的

异步：将耗时很长的A交付的工作交给系统之后，就去继续做B交付的工作。等到系统完成前面的工作之后，再通过回调或者事件，继续做A剩下的工作。

从观察者的角度来看，AB工作的完成顺序，和交付给他们的时间顺序无关，所以叫”异步”。

异步操作的常见语法：
ajax请求、jQuery监听页面加载之后的回调$(function(){})

浏览器中的JavaScript：

  1. 异步操作以事件为主 onload、ready等
  2. 回调主要出现在Ajax和File API
    有了Node.js之后，对异步的依赖进一步加剧了：
  3. 无阻塞高并发，是Nodejs的招聘，异步操作是其保障，大量操作依赖回调函数
    稍有不慎，就会踏入”回调地狱”。

异步回调有四个问题：

  1. 嵌套层次很深，难以维护
  2. 无法正常使用return和throw
  3. 无法正常检索堆栈信息
  4. 在多个回调之间难以建立联系

3.Promise详解

  1. Promise是一个代理对象，它和原先要进行的操作并无关系
    1. 它通过引入一个回调，避免更多的回调
    2. Promise有3个状态：
      1. Pending实例化时，【待定】状态
      2. 执行完毕，调用resolve时，fullfilled【实现】状态
      3. rejected【被否决】操作失败
        Promise状态发生改变，就会触发.then()里的响应函数处理后续步骤。
        Promise状态一经改变，不会再变。
  1. Promise实例一经创建，执行器立即执行；
  2. new 一个Promise传入执行器，创建实例后执行器开始执行，执行器是一个异步操作，接下来，通常会有后续的操作，调用第一个then，第二个then，最终生成一个队列，每一个then都会返回一个新的Promise实例。
  3. 当executor执行完毕之后，开始判断状态，把状态改变为成功或失败，接着调用then的对应的处理函数，一个then处理完了之后，循环到下一个then，反复的循环，直到队列当中所有的待处理的函数都执行完
  • Promise实现一个定时执行的小程序
  • 假如一个Promise已经完成了，再.then()会怎样？

我们在任何一个地方生成一个Promise队列之后，可以将它作为一个变量传递到其他地方，如果我们的操作是一个很明显的队列的先进先出的一个状态，我们就可以在它的后面去追续任意多的then，
不管前面的Promise是否完成，队列都会按照它对应的顺序去执行，如果没有完成，他就会顺序的完成，如果已完成，后面追加的then也会得到前面Promise返回的值。
这就是Promise身为队列，最重要的特性所在。

  • 假如在.then()的函数里面不返回新的Promise，会怎样？

Because：

中没有返回一个新的值，返回的Promise实例是在（3）中返回的，而不是在（2）的响应函数中返回的，
因此没有等待里面的return而直接return false，继续下一个then。
与此同时，（3）Promise中的timeout仍在等待执行，因此2s之后，输出了”Miss lina””
:在Promise中如果不直接返回一个Promise实例，它就会默认地去执行下一个环节，即使返回false也不会影响下一步。
如果注释：// return false;会返回 undefined world，没有明确返回值时就相当于return null。

  • 线程整个进程的进度，是在所有的都完成之后才会进行的，（3）中的Promise虽然没有进入到队列当中，没有终止2秒钟才能继续执行，但整个进程，是等他完成之后才宣告完成。

webpack学习03-核心概念(下)

sourceMap的配置

sourceMap是shong么？

当我们代码写错后，打包运行，在console控制台会有错误提示，但提示的是错误在打包之后的文件位置，而开发阶段，最重要的是错误定位到原文件，

sourceMap是一个映射关系，知道dist/main.js文件出错位置在原文件index.js文件中的位置。

sourceMap使用

在webpack.config.js中配置devtool属性为devtool: 'source-map',而这个是会影响性能
具体查看官网,使用devtool其实就是在使用sourcemap，配置之后，官网表格会显示打包的过程build、rebuild是会变慢的，因为打包过程中要构建这种映射关系.
打包运行后，dist目录下会生成main.js.map文件，其实就是打包后的文件与原文件的映射关系文件。

官网中：cheap、eval、inline、module等都可以自由组合，产生不同的打包效果：
devtool: 'inline-source-map'与devtool: 'source-map'区别，第一种无map文件，map文件会以dataURL的方式直接写在dist/main.js中。main.js.map文件会被变成base64对的字符串放在main.js的底部。
devtool: 'cheap-inline-source-map'与上述的区别，上述会告诉哪一行哪一列即哪个字符发生错误，这是比较耗费性能的，且我们实际只需要知道在哪一行就行，cheap会使打包性能得到一定的提升。
且cheap只对业务代码做映射，对loader等第三方引入的模块文件不做映射，而module会管loader、第三方模块中的错误。
eval是打包速度最快的一种方式，通过eval的执行方式，来生成sourcemap的对应关系，但是针对复杂的代码，该方式的提示可能不够全面。

最佳实践

开发环境mode：development：devtool:'cheap-moudle-eval-source-map'
线上环境mode：production：devtool:'cheap-module-source-map'

自2015年ECMAScript6发布后，新的特性及语法可以更方便的使用在我们的开发上面，更在每年都推出新ES规范，方便简洁，可以更优雅的开发跟js有关的应用。
但是浏览器的支援跟不上js版本的更新，所以新版本的js在现有的浏览器上面支援度很低，目前市面上支援度最高的版本大概就是ES5了，为了要让我们可以使用新版本的js做开发又要浏览器支援，
我们就需要有相应的工具做转换，把新版本的js转成ES5.

webpack作为打包工具的使用，本身并没有提供把新版版的js转换成ES5的功能，所以要靠转换工具babel。

什么是「babel」?

babel是一个目前被广泛使用的编译工具，可以运行在许多自动化工具上，我们可以使用webpack+babel来把新版本的js编译成ES5。

babel安装

安装babel（最新babel7）：
npm install babel-loader @babel/core @babel/preset-env --save-dev
安装完成后打开package.json文件：

babel-loader:webpack要去读取任何文件都要靠loader这个工具去做判断，然后再去做转换，所以我们就需要webpack挂载这个babel-loader。
@babel/core: 文件需要调用babel的API进行编译。
@babel/preset-env：可以使用最新版本的JavaScript然后去编译，不用去理会哪些语法需要转换。

总结

babel不只是转换新版本的js，也可以转换react的jsx、yscript等。

webpack学习01-项目搭建

虽然网上关于webpack项目的初始化搭建已经多的数不过来了，但还是要记录总结一下，毕竟自己写的，才是自己的知识。

1. webpack基本概念

随着前端技术的不断发展，简单的html、css、js早已不能满足我们的开发需求，很多前端工具和框架不断地冒出来，提高前端门槛，es6,less,sass,react等是我们目前开发所必需的，那么问题来了：浏览器
无法识别。

其实webpack帮我们做的事情很简单，就是帮我们把我们编写的preprocess（预处理）成浏览器看得懂的内容，然后打包成一包的完成档案，然后拿去server上传上去。

2. webpack工程初始化

2.1 搭建步骤

首先webpack是在nodejs上面运行的，所以电脑一定要安装nodejs.
然后在自己电脑中新建一个文件夹（不要webpack这种名称,安装npm webpack时会失败），然后进入该文件夹，npm init创建一个package.json
通过nodejs的套件管理工具npm来安装webpack。
npm install webpack webpack-cli --save-dev。

node_modules:我们通过npm下载下来的套件跟工具都会放在这个文件夹里；
package.json:包含安装的套件版本、npm指令、工程版本等都可以在json文件里找到，之后的工程搬移重新安装套件也都依靠这个json文档；
package-lock.json:是npm5新增的，是专门记录package.json里面的更细节的内容，例如安装的套件的详细版本，或是确认你的dependency（依赖）是被哪个库所要求的等等。

1.npm init //电脑环境配置nodejs/npm
2.安装webpack // 操作命令执行
3.配置webpack.config.js文件

4.增加文件入口src/index.js文件，写入内容
5.新增index.html引入文件入口，
6.package.json中增加script命令，执行webpack打包命令

终端执行命令npm run build即可看到打包后的dist文件夹中有index.bundle.js编译后的文件。

文件入口、出口路径配置

首先，webpack是建立在nodejs上面的工具，webpack只负责编译，别的需要靠nodejs的api来帮忙。

nodejs要搬移任何的文档都要靠一个叫path的api来做，所以在webpack.config.js中用require的方式来引入path的模块。
然后在module.exports里一开始加入context这个属性，里面使用path.resolve这个方法来指定一开始的路径。
context: path.resolve(__dirname, ‘src’),

__dirname在nodejs里代表的一个特殊变量，指的是当前执行文件所在目录的完整目录位置，然后指定第二个参数，从src资料夹开始找文件。
如果想输出不是在dist文件夹内，可以在output中重新指定路径：

watch指令可以帮助我们监控js，只要有js文件有变动就会自动帮我们执行编译，执行相关watch命令后，会发现命令行中webpack虽然编译了，但是执行却没有中断，因为webpack是watch的状态。
mode未设置：意思就是webpack在执行编译的时候有production（上线版本，需要将档案给压缩和优化）和development（开发中，只编译）状态。

webpack4默认是production，所以在不设定mode的情况下watch会执行production功能，但是这样不太好，因为开发中的时候你每次执行watch就除了帮你编译还执行压缩和优化，太消耗资源，而且会占太多电脑记忆，所以日常开发使用development模式。
避免在执行的时候消耗不必要的资源。

2.2 目录结构
  1. 项目生成后有两个很重要的文件夹，src和dist。
    （1）src是专门存放我们的资源文件的，包括es6、less、jsx等文件，这些文件夹不会被部署到server上；
    （2）dist是经过webpack编译打包后，产生出的浏览器看得懂的html、css、js，这些文件会被部署到server上。

3. 相关概念

什么是「注入」?

webpack是一个以自己编写设定档案然后透过指令去驱动的一个自动化工具，所有的动作都是透过自己写的规则去做编译，我们必须透过JavaScript
的引入语法让webpack知道需要它帮忙编译什么东西，例如less、pug等。
所以我们会有一支俗称进入点的js专门注入那些Preprocess，让那些Preprocess可以通过这个进入点的js让webpack去编写它，进而打包到dist资料夹。
负责项目越大时，需要注入的东西也越来越多，如jsx、less、eslint、babel等。

4. 总结

不管什么工具都是为了帮助你在开发工作上更顺利，重点还是前端能力的基础。

websocket机制

1.引言

在对HTTP协议有了简单了解之后，结合之前的项目，项目中有时客户端需要实时的获得服务器消息，但默认HTTP协议只支持请求响应模式，这种处理方式都会为增加服务器带宽，为其带来压力，在进行数据轮询时还要经常考虑服务器是否能承受。
websocket，支持客户端和服务器端的双向通信，可以简化Web服务器，减少服务器的负担，加快响应速度。

2.什么是websocket

websocket是HTML5提出的一个协议规范，websocket约定了一个通信的规范，通过握手机制，客户端和服务器就能建立一个类似TCP的连接，从而方便客户端和服务器的通信。

WebSocket特点：

  1. 服务器可以主动向客户端推送信息，客户端也可以主动向服务器发送信息，是真正的双向平等对话
  2. 建立在 TCP 协议之上，服务器端的实现比较容易
  3. 与 HTTP协议有着良好的兼容性。默认端口也是80和443，并且握手阶段采用 HTTP协议，因此握手时不容易屏蔽，能通过各种HTTP代理服务器
  4. 数据格式比较轻量，性能开销小，通信高效
  5. 可以发送文本，也可以发送二进制数据
  6. 没有同源限制，客户端可以与任意服务器通信
  7. 协议标识符是ws（如果加密，则为wss），服务器网址就是 URL

3.如何使用

websocket连接的流程：建立一个websocket链接，客户端或者说浏览器首先向服务器发起一个HTTP请求，这个请求和普通的HTTP请求不同，他包含了一些附加头信息，其中呢，就有一个头信息叫做upgrade websocket，服务器端会解析这些附加头信息， 然后生成应答消息响应给客户端，客户端呢与服务器端呢就建立了相应的链接。

WebSocket 在服务端的实现非常丰富。Node.js、Java、C++、Python 等多种语言都有自己的解决方案。

常见x，y属性

1.引言

对常见CSS x,y属性有大概了解，实际运用过程中发现很多概念模糊不清，对其进行学习。

2.常见x,y属性概念

图解如下：

在不同浏览器下，各种属性的定义不同，表现也有细微差异，因此 推荐使用: screenX/Y、pageX/Y、clientX/Y、offsetX/Y 此外: X/Y与clientX/Y相同，firefox不支持 layerX/Y除IE外与pageX/Y相同，IE11下与clientX/Y相同，非官方属性。

3.相关scroll、offset属性概念

scrollTop是指某个可滚动区块向下滚动的距离。
offsetTop则是元素的上边框与父元素的上边框的绝对距离。

  1. offsetTop:
    当前对象到其上级层顶部的距离.不能对其进行赋值.设置对象到页面顶部的距离用style.top属性.
    （style.top属性必须自己设置，系统默认为空,设置方式：
    document.getElementById(“box”).style.top=”100px”;）
  2. offsetLeft:
    当前对象到其上级层左边的距离.不能对其进行赋值.设置对象到页面左部的距离用style.left属性.
  3. offsetWidth:
    当前对象的宽度.与style.width属性的区别在于:如对象的宽度设定值为百分比宽度,则无论页面变大还是变小,style.width都返回此百分比,而offsetWidth则返回在不同页面中对象的宽度值而不是百分比值
  4. offsetHeight:
    与style.height属性的区别在于:如对象的宽度设定值为百分比高度,则无论页面变大还是变小,style.height都返回此百分比,而offsetHeight则返回在不同页面中对象的高度值而不是百分比值
  5. offsetParent:
    HTMLElement.offsetParent 是一个只读属性，返回一个指向最近的（closest，指包含层级上的最近）包含该元素的定位元素。如果没有定位的元素，则 offsetParent 为最近的 table, table cell 或根元素。当元素的 style.display 设置为 “none” 时，offsetParent 返回 null。offsetParent 很有用，因为 offsetTop 和 offsetLeft 都是相对于其内边距边界的。
    offsetParent属性 (1) 当某个元素及其DOM结构层次中元素都未进行CSS定位时(absolute或者relative)[或者某个元素进行CSS定位时而DOM结构层次中元素都未进行CSS定位时]，则这个元素的offsetParent属性的取值为根元素。更确切地说，这个元素的各种偏移量计算（offsetTop、offsetLeft等）的参照物为Body元素。(其实无论时标准兼容模式还是怪异模式，根元素都为Body元素) (2) 当某个元素的父元素进行了CSS定位时（absolute或者relative），则这个元素的offsetParent属性的取值为其父元素。更确切地说，这个元素的各种偏移量计算（offsetTop、offsetLeft等）的参照物为其父元素 (3) 当某个元素及其父元素都未进行CSS定位时（absolute或者relative），则这个元素的offsetParent属性的取值为在DOM结构层次中距离其最近，并且已进行了CSS定位的元素。
  6. scrollLeft:
    对象的最左边到对象在当前窗口显示的范围内的左边的距离．
    即是在出现了横向滚动条的情况下，滚动条拉动的距离．
  7. scrollTop
    对象的最顶部到对象在当前窗口显示的范围内的顶边的距离．
    即是在出现了纵向滚动条的情况下，滚动条拉动的距离

相关属性

DOM innerheight、innerwidth属性:
只读属性，声明了窗口的文档显示区的高度和宽度，以像素计。
这里的宽度和高度不包括菜单栏、工具栏以及滚动条等的高度。

IE 不支持这些属性。它用 document.documentElement 或 document.body的 clientWidth 和 clientHeight 属性作为替代。

pageXOffset 和 pageYOffset 属性返回文档在窗口左上角水平和垂直方向滚动的像素。pageXOffset 和 pageYOffset 属性相等于 scrollX 和 scrollY 属性。这些属性是只读的。
pageYOffset 属性是 scrollY 属性的别名：window.pageYOffset == window.scrollY;
window.pageYOffset == window.scrollY
所有主流浏览器都支持 pageXOffset 和 pageYOffset 属性。
但是 IE 8 及 更早 IE 版本不支持该属性,但可以使用 document.documentElement.scrollLeft 和 document.documentElement.scrollTop 属性 。
offsetTop：
获取任意元素，距离页面顶部（不是文档显示区）的距离
innerHeight:文档显示区的高度

scrollTop：页面顶部超出文档显示区的距离
scrollLeft = pageXOffset = scrollX 都代表向左滚动偏移X轴的距离，Y同理为向上滚动偏移距离，兼容不同浏览器而做的处理。

4.使用场景

例如本周项目中遇到：当scrollTop+innerHeight < offsetTop时,DOM1未在可视窗口显示，
scrollTop+innerHeight > offsetTop时,DOM1已经滚动至可视窗口内，
根据此可以监听元素在可视窗口区内滚动显示范围，从而可以进行“回到顶部”按钮添加判断，以及对当前加载DOM元素进行动态加载效果添加等处理。
