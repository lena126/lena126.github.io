HTML5调起APP应用程序

1.前言

在项目中，通过HTML5实现APP应用程序开发时，需要完成以下需求：判断当前用户是否下载APP，已下载：跳转至APP内，未下载：去下载。目前客户端同学不支持对用户手机内是否有该APP做出判断，因此采用以下思路：提供两个button按钮，【已下载，去首页】【未下载，去下载】，这样让用户手动判断是否下载，避免用户已下载重复下载以及跳转登录页判断执行的矛盾。

2.优化

目前思路能够满足功能需求，但是对于一些用户未下载，点击已下载这种情况的恶意点击操作，以及部分手机浏览器对于window.location.href跳转支持的兼容性，仍需做出跳转失败判断及提示。

3.解决思路

  1. 设置一个延迟执行提示，然后判断当前页面是否被hidden
  2. 被hidden证明跳转成功，清楚定时器
  3. 几秒之后仍旧没有被hidden证明跳转失败，做出提示信息

4.总结

关于判断从手机浏览器跳转至指定APP是否成功，需要移动端相关支持，浏览器无法明确地判断本地是否已经安装 app.以及Android、iOS不同情况，以及手机端华为、三星、UC、谷歌等浏览器较多，兼容性情况不好判断，且实现起来较复杂，这是目前想到的比较完善的操作提醒。当然，只是简单的调研实现，肯定有更优秀的方法需要去学习发现。

学习周报2019-08-09

1.保存图片至本地

下载文件是一个非常常见的需求，前端保存页面为图片，最常见的方案都是利用html2canvas来生成图片，同时利用HTML5的新属性download来点击保存到本地。

2.PC浏览器下载图片

网上一些实现方法由于跨域会导致a标签在部分浏览器中会直接打开新标签页，因此最终实现兼容性较好的下载方式为用 js 去获取图片，获得其 base64，放回 img 的 src 中再进行绘制。

分为四步：

  1. 用 JavaScript 创建一个隐藏的a标签
  2. 设置它的 href 属性
  3. 设置它的 download 属性
  4. 用 JavaScript 来触发这个它的 click 事件

跨域问题需要在给 img 元素设置 crossOrigin:anonymous;属性，图片路径服务端必须设置允许跨域。
为了更快的响应，很多 CDN 会缓存图片的返回值，而缓存的值是不带 CORS 头的。因为没有 CORS 头，所以 js 请求会被拦截。
因此本地文件及图片管理系统中的图片可以下载，网络在线图片及iPic上的图片链接均存在跨域问题无法下载。
（除img标签图片下载之外还有canvas图片下载等方法。）

3.通过HTML5实现APP保存图片至相册

通过HTML5实现APP保存图片，需要调用手机系统中相册/保存图片到相册，经过查阅发现，现有APP保存图片HTML相关方法是通过HTML5 plus即中国HTML5产业联盟的扩展规范来进行实现。
通过属性plus.gallery获取相册管理对象。
具体开发实现参考：HTML5 Plus实现APP图片相关操作

4.总结

PC浏器实现下载图片该方法是基于从 DOM 读取的元素及属性来一点点的绘制 canvas，因此，它只能正确地呈现它理解的元素和属性，这意味着有许多 CSS 属性不起作用。图片大小、图片路径跨域问题、图片格式等都可能导致下载不成功。通过原生HTML5以及JS实现保存图片至APP本地相册并未找到方法实现。关于保存图片，涉及之前没了解过的知识点较多，关于相关学习本次只是简单了解，具体掌握还需继续深入学习。

1.WEEX概念

SBridge 简单来讲，主要是 给 JavaScript 提供调用 Native 功能的接口，让混合开发中的『前端部分』可以方便地使用地址位置、摄像头甚至支付等 Native 功能。
既然是『简单来讲』，那么 JSBridge 的用途肯定不只『调用 Native 功能』这么简单宽泛。实际上，JSBridge 就像其名称中的『Bridge』的意义一样，是 Native 和非 Native 之间的桥梁，它的核心是 构建 Native 和非 Native 间消息通信的通道，而且是 双向通信的通道。

2.word-break属性

word-break指定了在单词内如何换行。word-break有以下四种常见属性值：

运用到元素中，从下图看出，同样的文案，在不同的属性下，折行效果不同：

（1）work-break:break-all；经常用到，对于non-CJK (CJK 指中文/日文/韩文) 文本，可在任意字符间断行，第一个div中文案到达右边框后都被强制折行显示；

（2）work-break:break-word；从第二个div中可以看出，上面url部分，折行部分与break-all的区别在于二行折行部分，即该属性会对英文单词先起新行，放不下再强制断行；
像URL这种长字符串，“？-”这些符号会被识别作为一个单词的结束，进行断行；

（3）work-break:normal；只在允许的断字点换行（浏览器保持默认处理），允许的断字点图中看出”-“，”?”属于。如果是中文则到边界处的汉字换行，如果是英文整个词换行，如果出现某个英文字符串长度超过边界，则后面的部分将撑开边框，如果边框为固定属性，则后面部分将溢出；

（4）work-break:keep-all；只能在半角空格或连字符处换行。不允许字断开，如果是中文将把前后标点符号内的一个汉字短语整个换行,英文单词也整个换行，如果出现某个英文。字符串长度超过边界，则后面的部分将撑开边框，如果边框为固定属性，则后面部分将溢出；

3.类似属性​

word-wrap:用来标明是否允许浏览器在单词内进行断句，这是为了防止当一个字符串太长而找不到它的自然断句点时产生溢出现象；

word-break:用来标明怎么样进行单词内的断句；

white-sapce:用于处理元素内的空白，只在一行内显示；

4.总结

一般中文超出文本区域都会自动换行，但是对于英文折行显示，无论如何单词内的断句都会都单词的可读性产生一定的影响，如果不进行单词断句，每行后长短不一的空白，又会对整个文本区域的美观造成影响。
因此一般处理方式：
（1）强制换行，保证每行的布局完整，同时在强制断开的单词处添加连词符号，确保直观上的单词完整性；

（2）文本在固定区域显示，会增加属性

控制文本显示指定行数，超出…显示，防止文本数据长短不可控，过分长超出显示区域；
对于URL等外文特殊长字符串，尽量采用word-break:break-all;保持界面布局整洁性。
