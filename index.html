promise学习03-Promise.all

Promise.all()

批量执行一堆函数

  1. Promise.all([p1,p2,p3,…])用于将多个Promise实例，包装成一个新的Promise实例。
  2. 返回的实例就是普通Promise。
  3. 它接收一个数组作为参数
  4. 数组里可以是Promise对象，也可以是别的值，只有Promise会等待状态的改变
  5. 当所有子Promise都完成，该Promise完成，返回值是全部值得数组
  6. 有任何一个失败，该Promise失败，返回值时第一个失败的子Promise的结果。

执行结果：

  1. (1)中传入的是[1,2,3]，不是数组，结果会被立刻返回
  2. (2)中传入的是function，不是Promise实例，也会被立刻执行
  3. （5)中接收到的都是Promise实例，因此会等待1.5s左右返回两个Promise(3)(4)的返回值组成的结果
  4. (8)有p1,p2,p3，同时p2,p3都会返回错误，p2会错误的早一些，所以(9)会被跳过，同时(10)catch触发，捕获到错误，是由第一个触发错误的Promise返回的值(7)

Promise.all() 和.map()使用

Promise.all()最常见的就是和.map()，把一组操作转化成Promise集体的状态。

实现队列

有时候我们不希望所有动作一起发生，而是按照一定顺序，逐个进行。
通过Promise.then()返回一个新的Promise实例的特性，串成队列来进行操作。
如果有一组要做的事，有两种方式：
（1）使用.forEach()、for循环

（2）使用.reduce() 从数组的一端遍历数组，直到另外一端

两个常见错误：

（1）中与forEach连用时，没有把.then()产生的新Promise实例赋给Promise，没有生成队列，造成当Promise完成时，后面所有的then会同时触发
（2）中与reduce连用时，Promise实例创建之后，会立刻运行执行器代码，所以所有的Promise也会立刻执行，不会等待队列的下一步才执行。

promise学习02-then

then的执行效果

  1. .then()接受两个函数作为参数，分别代表fulfilled下的响应函数和rejected下的响应函数
  2. .then()返回一个新的Promise实例，所以它可以链式调用
  3. 当前面的Promise状态改变时，.then()根据其最终状态，选择特定的状态响应函数执行
  4. 状态响应函数可以返回新的Promise，或其他值
  5. 如果返回新的Promise，那么下一级.then()会在新Promise状态改变之后执行
  6. 如果返回其他任何值，则会立刻执行下一级then

1. .then的嵌套

.then()里有.then()的情况
  1. 因为.then()返回的还是Promise实例
  2. 会等里面的.then()执行完，再执行外面的
  3. 对我们来说，此时最好将其展开，会更好读。

3. 错误处理

Promise会自动捕获内部异常，并交给rejected响应函数处理。

只能捕获异步完成回调之后的错误信息，在Promise的执行器中如果发生错误，那么Promise的状态就会被置为rejected，后面fulfilled的响应函数不会被执行，而
后面通过catch来捕获错误的函数就会被执行。

在Promise中错误处理有两种做法：
  1. reject('错误信息').then(null,message=>{})
  2. throw new Errow('错误信息').catch(message =<{})
    推荐使用第二种，更加清晰好读，并且可以捕获前面的错误。
.catch()+.then()会发生什么？

catch也会返回一个Promise实例，并且如果其中没有抛出错误这个Promise实例也是fulfilled的状态。

而我们通过捕获，把错误处理一下或记录下来，都会有比较好的效果。

promis学习

1.promise是什么

MDN给出的定义：Promise对象用于异步计算，一个Promise表示一个现在、将来或永不可能可用的值。
按用途来解释：

  1. 用于异步计算
  2. 可以将异步操作队列化，按照期望的顺序执行，返回符合预期的结果。
  3. 可以在对象之间传递和操作Promise，帮助我们处理队列。

2.为什么会有promise

  1. JavaScript为检查表单而生
  2. 创造它的首要目标是操作DOM
  3. 所以，JavaScript很多操作是异步的

异步：将耗时很长的A交付的工作交给系统之后，就去继续做B交付的工作。等到系统完成前面的工作之后，再通过回调或者事件，继续做A剩下的工作。

从观察者的角度来看，AB工作的完成顺序，和交付给他们的时间顺序无关，所以叫”异步”。

异步操作的常见语法：
ajax请求、jQuery监听页面加载之后的回调$(function(){})

浏览器中的JavaScript：

  1. 异步操作以事件为主 onload、ready等
  2. 回调主要出现在Ajax和File API
    有了Node.js之后，对异步的依赖进一步加剧了：
  3. 无阻塞高并发，是Nodejs的招聘，异步操作是其保障，大量操作依赖回调函数
    稍有不慎，就会踏入”回调地狱”。

异步回调有四个问题：

  1. 嵌套层次很深，难以维护
  2. 无法正常使用return和throw
  3. 无法正常检索堆栈信息
  4. 在多个回调之间难以建立联系

3.Promise详解

  1. Promise是一个代理对象，它和原先要进行的操作并无关系
    1. 它通过引入一个回调，避免更多的回调
    2. Promise有3个状态：
      1. Pending实例化时，【待定】状态
      2. 执行完毕，调用resolve时，fullfilled【实现】状态
      3. rejected【被否决】操作失败
        Promise状态发生改变，就会触发.then()里的响应函数处理后续步骤。
        Promise状态一经改变，不会再变。
  1. Promise实例一经创建，执行器立即执行；
  2. new 一个Promise传入执行器，创建实例后执行器开始执行，执行器是一个异步操作，接下来，通常会有后续的操作，调用第一个then，第二个then，最终生成一个队列，每一个then都会返回一个新的Promise实例。
  3. 当executor执行完毕之后，开始判断状态，把状态改变为成功或失败，接着调用then的对应的处理函数，一个then处理完了之后，循环到下一个then，反复的循环，直到队列当中所有的待处理的函数都执行完
  • Promise实现一个定时执行的小程序
  • 假如一个Promise已经完成了，再.then()会怎样？

我们在任何一个地方生成一个Promise队列之后，可以将它作为一个变量传递到其他地方，如果我们的操作是一个很明显的队列的先进先出的一个状态，我们就可以在它的后面去追续任意多的then，
不管前面的Promise是否完成，队列都会按照它对应的顺序去执行，如果没有完成，他就会顺序的完成，如果已完成，后面追加的then也会得到前面Promise返回的值。
这就是Promise身为队列，最重要的特性所在。

  • 假如在.then()的函数里面不返回新的Promise，会怎样？

Because：

中没有返回一个新的值，返回的Promise实例是在（3）中返回的，而不是在（2）的响应函数中返回的，
因此没有等待里面的return而直接return false，继续下一个then。
与此同时，（3）Promise中的timeout仍在等待执行，因此2s之后，输出了”Miss lina””
:在Promise中如果不直接返回一个Promise实例，它就会默认地去执行下一个环节，即使返回false也不会影响下一步。
如果注释：// return false;会返回 undefined world，没有明确返回值时就相当于return null。

  • 线程整个进程的进度，是在所有的都完成之后才会进行的，（3）中的Promise虽然没有进入到队列当中，没有终止2秒钟才能继续执行，但整个进程，是等他完成之后才宣告完成。

sourceMap的配置

sourceMap是shong么？

当我们代码写错后，打包运行，在console控制台会有错误提示，但提示的是错误在打包之后的文件位置，而开发阶段，最重要的是错误定位到原文件，

sourceMap是一个映射关系，知道dist/main.js文件出错位置在原文件index.js文件中的位置。

sourceMap使用

在webpack.config.js中配置devtool属性为devtool: 'source-map',而这个是会影响性能
具体查看官网,使用devtool其实就是在使用sourcemap，配置之后，官网表格会显示打包的过程build、rebuild是会变慢的，因为打包过程中要构建这种映射关系.
打包运行后，dist目录下会生成main.js.map文件，其实就是打包后的文件与原文件的映射关系文件。

官网中：cheap、eval、inline、module等都可以自由组合，产生不同的打包效果：
devtool: 'inline-source-map'与devtool: 'source-map'区别，第一种无map文件，map文件会以dataURL的方式直接写在dist/main.js中。main.js.map文件会被变成base64对的字符串放在main.js的底部。
devtool: 'cheap-inline-source-map'与上述的区别，上述会告诉哪一行哪一列即哪个字符发生错误，这是比较耗费性能的，且我们实际只需要知道在哪一行就行，cheap会使打包性能得到一定的提升。
且cheap只对业务代码做映射，对loader等第三方引入的模块文件不做映射，而module会管loader、第三方模块中的错误。
eval是打包速度最快的一种方式，通过eval的执行方式，来生成sourcemap的对应关系，但是针对复杂的代码，该方式的提示可能不够全面。

最佳实践

开发环境mode：development：devtool:'cheap-moudle-eval-source-map'
线上环境mode：production：devtool:'cheap-module-source-map'

webpack-dev-server不生成dist目录，会放在电脑的内存中，会有效的提升打包的速度，让开发更快，webpackdev-server隐藏的一个目录。

  1. scripts:”webpack-dev-server”命令运行后，打包不会生成dist目录，会帮助我们启动一个localhost服务器。
    1. webpack-dev-server简介：
      是一个小型node.js express服务器
      新建一个开发服务器，可以serve我们pack以后的代码，并且当代码更新的时候自动刷新浏览器
      启动webpack-dev-server后，你在目标文件夹中是看不到编译后的文件的，实时编译后的文件都保存到了内存当中。
    2. 两种自动刷新方式：
      1. iframe mode
        在网页中嵌入了一个 iframe ，将我们自己的应用注入到这个 iframe 当中去，因此每次你修改的文件后，都是这个 iframe 进行了 reload
        命令行：webpack-dev-server，无需–inline
        浏览器访问：http://localhost:8080/webpack-dev-server/index.html
        inline mode
      2. 命令行：webpack-dev-server –inline
        浏览器访问：http://localhost:8080
    3. 安装webpack-dev-server
      npm install webpack-dev-server –save-dev
      可在项目根目录下安装（不加-g，在项目根目录出现node_modules文件夹，内含webpack-dev-server及其依赖包），也可全局安装（加-g 必须sudo）
